## HTTPS

### HTTP
	⁃	정의 : 웹 서버 & 클라이언트(브라우저)가 데이터를 주고받는 통신 규약(프로토콜)
	⁃	단점 : 평문(plain text)로 전송하기 때문에 보안에 취약함
 
### HTTPS
	⁃	HTTP Secure : HTTP + SSL/TLS 보안 계층을 추가
	⁃	장점 : 보안에 좋음(대칭키, 비대칭키, 인증서 등 사용) 

### HTTP 암호화 방식(2) : 대칭키, 비대칭키
	⁃	대칭키 : 세션키 사용 - 하나의 키로 암호화/복호화 가능. (단점 : 대칭키 해킹되면 정보털림) 
	⁃	비대칭키 : 공개키 + 개인키 사용(A 키로 잠구면 B키로 열 수 있음.  or B키로 잠구면 A키로 열수 있음.)  

### 비대칭키
	⁃	공개키(A) : 모두가 쓸 수 있게 뿌려진 공개된 키
	⁃	개인키(B) : 하나만 들고 있는 개인 키 = 공개키(A) 열수 있는 1개의 키( B) 
	⁃	장점 : 개인키(B)는 서버에서 1개만 갖고있기 때문에 해킹 못함.(보안 굿)
	⁃	단점 : 가짜서버(해커)에서 공개키(A’) 뿌리면 클라이언트는 어떤게 진짜 공개키(A)인지 모름 >> 인증서로 검증  

### 인증서 
	⁃	생긴 이유 : 서버(B)의 공개키(A)가 맞는지 검증하기 위함 (내가 접근하고자 하는 서버(B)에 맞는 공개키가 A인지 A’인지 검증 - 보안 up) 
	⁃	TLS 핸드쉐이크 : 본 암호화 통신 전, 서버 진짜 여부 확인 
	⁃	검증 원리 :
	0.	서버 -> CA(인증기관)으로 사이트 정보+공개키(A) 전달
	0.	CA : CA의 개인키(C)로 서명(암호화) 해서 -> 인증서 발급해줌(CA에서만 풀 수 있음, 공개키(D))
	0.	클라이언트가 -> 서버로 인증서 요청(Client Hello)
	0.	서버 -> 클라이언트에게 인증서 전달(Server Certification)
	0.	클라이언트 : 인증서 -> CA에 검증요청
	0.	CA : 인증서(공개키(D))가 CA의 개인키(C)로 풀리면 검증 완료 - 이 안에는 사이트정보+공개키(A) 들어있음. 
	0.	클라이언트 : 인증서 안에 있던 서버의 사이트정보 + 공개키(A)로 서버와 정보 전달
	0.	이후 클라이언트(공개키A) & 서버(개인키B) : 안전하게 정보 전달 가능

## 비동기/동기
### Q1. 비동기/동기는 언제 적용하는가?
	⁃	동기 : 계산(CPU) 작업인가? -> for문 반복, 이미지처리, AI학습, 복잡한 연산
	⁃	비동기 : 대기 작업인가? -> 대표적으로 I/O작업(API호출, DB조회, 파일 읽기, LLM호출)

### Q2. A작업(5초), B작업(1초)이면 결국 총 시간 6초 아닌가? CPU는 병렬처리 불가하지 않나?
	⁃	계산 작업이 아니고, 동시에 A, B작업이 이루어진다면 5초 안에 처리 가능.
	⁃	비동기의 핵심은 시간을 줄이는 게 아니라 기다리는 시간을 겹쳐 쓰는 것이다. 
	⁃	비동기는 작업 자체를 병렬로 처리하는 기술이 아니라,  외부 응답을 기다리는(I/O) 동안 CPU를 다른 작업에 사용하도록 만들어 전체 처리 효율과 동시 처리량을 높이는 방식이다.

### Q3. 비동기는 그럼 CPU가 가동하지 않는 시간 = await 가능한 시간인가?
	⁃	현재 작업이 끝날 때까지, 다른 작업 해도 되는 시간 = await 후보

# LLM 설치 과정에서 AI 물어본 것

### curl 이란?
	⁃	터미널에서 HTTP 요청을 보내는 도구(브라우저 없이 API 테스트하는 프로그램)

### httpx 란?
	⁃	Python에서 HTTP요청을 보내기 위해 사용하는 라이브러리. 
	⁃	비동기처리, 스트리밍 응답에 강점

### Request Body 랑 payload의 차이점이 뭐야?
	⁃	Request Body : HTTP 요청 구조 안의 "위치"
	⁃	Payload : 실제 전달되는 "데이터 내용" (데이터 알맹이)
	⁃	Body ⊃ Payload

### brew 와 pip 의 차이
	⁃	brew : 맥 컴퓨터에 프로그램을 설치 (brew install ollama -> ollama serve)
	⁃	pip : Python .venv 안에 라이브러리 설치 (pip install ollama -> import ollama)

### brew로 설치하면 내 맥에서 설치되서 ollama 효율이 떨어지는 건지?
	⁃	ollama는 그냥 LLM을 내 컴퓨터에 실행하게 해주는 프로그램(런타임/서버) 이다.
	⁃	ollama에서 모델(ex. Gemma)을 내려받아 추론하는 순간 맥 자원을 사용함
	⁃	모델 == 파라미터(가중치) 값 다운 받는 것 이라 용량 차지

### LLM 추론할 때 메모리에 영향주는 값들?
	⁃	LLM 추론 비용 : 파라미터 수 + 입력 길이(토큰 수) + Attention 계산량(토큰의 제곱)+ 출력 길이가 같이 결정
	⁃	보통은 짧은 입력 → 파라미터 수 영향이 큼, 긴 입력 → Attention(토큰 길이) 영향이 급격히 커짐


### Ollama 에게 요청할때, 우리가 클라이언트 입장에서 요청값을 쏴줘야하는데, 이때 필요한 요청값이나 구조는 어떻게 알 수 있어?
	⁃	클라이언트가 서버에 어떤 Header와 Body를 보내야 하는지는  서버가 제공하는 API 문서(OpenAPI/Swagger/공식 문서)에 정의되어 있다.

	⁃	클라이언트(httpx)는 그 규격대로 JSON을 만들어 보내고, 서버(Ollama)는 그 구조가 맞는지 검증한 뒤 처리한다.

	⁃	Ollama structured JSON : https://docs.ollama.com/capabilities/structured-outputs#python

	⁃	LLM api 연동할때 보통 순서: 주소 확인(url) -> 인증 확인(API key 필요한지) -> 요청 구조 확인(어떤 JSON?) -> 응답 구조 확인(어디서 content 꺼내지?)

### Q. TCP란?
	⁃	*TCP*(Transmission Control Protocol) : 컴퓨터끼리 데이터를 안정적으로 주고받기 위한 통신 규칙(프로토콜)
	⁃	실제로 컴퓨터에서는 데이터 옮길때 쪼개진 ‘패킷’ 단위로 움직이는데, 이때 패킷 유실, 순서 꼬임, 중복 일어날 수 있음. 이걸 TCP가 방지해줌.
	⁃	TCP(HTTP) <-> UDP(영상통화)
	⁃	TCP 연결유지(keep-alive) : TCP handshake로 만든 TCP 연결을 끊지않고 사용하는 것
